Claude Code AWS Bedrock Manager

Self-service management control plane for provisioning and managing Claude Code access via AWS Bedrock. Automates credential creation, approval, distribution, and lifecycle management.

Cost centre = the unit of budget ownership (can be a project, department, team, or any organisational unit).

User Roles

A user can hold multiple roles simultaneously (e.g., a Cost Centre Owner can also be a Developer with their own keys).

🛡️

Administrator

Governance, security, and organisational oversight. Acts as the ultimate authority.
  • Break-glass super admin (configured outside the UI, no UI needed)
  • Configures global platform settings (region, model access, policies)
  • Creates cost centres
  • Assigns Cost Centre Owners to cost centres
  • Can approve/reject any key request (bypassing Cost Centre Owner)
  • Views usage dashboards and cost reports across all cost centres
  • Manages users and their access
  • Can revoke any key
  • Sets global defaults for model restrictions and key policies
  • Global model removal revokes that model access from all existing keys
⌨️

Developer

Getting access to Claude Code quickly and with minimal friction. Just wants a working key.
  • Logs in to the self-service portal
  • One key per cost centre, multiple cost centres simultaneously
  • Requests a key by selecting from all non-archived cost centres
  • Views own active keys, usage, and limit status
  • Copies key + setup instructions (retrievable after creation)
  • Can revoke any of their own keys
  • Can see when stopped (limit reached) and when budget returns
  • Can resubmit after rejection (no limit); sees rejection reason
  • Keys auto-disabled on SSO deactivation (offboarding)
💰

Cost Centre Owner

Managing cost and usage within their cost centre(s). Owns the budget, decides who gets access and what limits apply.
  • Assigned by Administrator to one or more projects
  • A cost centre can have multiple CCOs; any single one can approve
  • Approves or rejects developer key requests
  • Own key requests auto-approved (if also a Developer on same cost centre)
  • Manages per-key constraints: cost limit (dollars), models, expiry
  • Can modify constraints on existing active keys (developer notified)
  • Can revoke keys for developers in their cost centre
  • Sets and manages Cost Centre Budget Cap (dollars); can top up
  • Can Archive cost centre (disables keys, auto-rejects pending requests)
  • Can unarchive own cost centre (restores keys for current org members)
  • Has scoped dashboard with usage, costs, alerts
  • Configures alerts (email and Slack — notifications only)
  • Can export CSV report for their cost centre(s)

Key Request Flow

End-to-end flow from request to working Claude Code access.

Happy Path

1
Developer

Opens portal, clicks "Request Key"

2
Developer

Selects a cost centre from all non-archived cost centres, optionally enters justification

3
System

If developer is also a CCO for that cost centre → auto-approved (skip to step 6). Otherwise, enters Pending Approval state.

4
System

Cost Centre Owner(s) for that cost centre are notified

5
Cost Centre Owner

Any single CCO reviews request, approves with constraints (or rejects with reason — developer can resubmit)

6
System

Automatically provisions the key, tagged with cost centre code for cost attribution (target: <10 seconds)

7
Developer

Notified, copies setup instructions, pastes into terminal, runs claude — it works immediately

8
System

Usage tracked against cost centre; if rolling period limit reached, key is stopped until budget becomes available

Admin Override Path

—
Administrator

Can approve any key request directly, bypassing Cost Centre Owner at any time

Rolling Window Mechanism

Cost is tracked over a sliding window of configurable days. As old days fall off the window, their spend frees up.

Functional Requirements

Platform capabilities organised by surface area.

Authentication System

  • Production: Corporate SSO (OIDC/SAML)
  • PoC: Hard-coded usernames and passwords
  • No external auth dependencies for PoC

Developer Dashboard Developer

  • View all active keys (one per cost centre, across multiple cost centres)
  • Key status: Active, Expired, Revoked, Stopped, Pending
  • Current spend vs limit within rolling period
  • When budget becomes available again if stopped
  • Usage metrics per key (tokens, cost)
  • Retrieve key credentials (re-copy setup instructions)
  • Revoke own keys

Request Management Admin

  • View all pending requests across all cost centres
  • Approve/reject any request (bypasses owner)
  • Bulk approve/reject

Cost Centre Management Admin

  • Create new cost centres
  • Assign/remove Cost Centre Owners
  • Deactivate cost centres (revokes all keys)
  • Unarchive a cost centre (restores keys for current org members; expired keys remain expired)

Key Management Admin

  • View all active keys across organisation
  • Revoke any key (triggers cloud cleanup)
  • Set global policies: max lifetime, models, rate limits
  • Search/filter by developer, cost centre, status

Cost & Usage Dashboard Admin

  • View aggregate usage and cost
  • Break down by cost centre code
  • Break down by developer/team
  • Set budget alerts per cost centre or globally
  • Export CSV report for a time period (cost per cost centre/cost centre)

Configuration Admin

  • Cloud account/region configuration
  • Allowed Claude models (Opus, Sonnet, Haiku)
  • Default key expiry and limit policies
  • Notification settings (Slack workspace, email server)

Approval & Key Config Owner

  • View pending requests for their cost centre(s)
  • Any single CCO can approve (consensus not required)
  • Approve with constraints (rolling cost limit, budget, models, expiry)
  • Reject with reason (developer can resubmit)
  • Modify constraints on existing keys (developer notified)
  • Revoke keys in their cost centre
  • Archive cost centre (disables keys, auto-rejects pending requests)
  • Unarchive own cost centre (restores keys for current org members)

Cost Centre Usage View Owner

  • Developer activity table (keys, status, spend vs limits)
  • Model usage breakdown per developer
  • Token consumption (input/output/total)
  • Cost tracking and trends over time
  • Charts: usage trends, burn-rate, model mix, top consumers
  • Budget management: cap (dollars), remaining, % used; can top up
  • Export CSV report for their cost centre(s)

Alert Configuration Owner

  • Email and Slack (channel or DM) — notifications only, no approval workflows
  • cost centre budget threshold alerts (50%, 80%, 100%)
  • Per-developer threshold alerts
  • Unusual usage spike alerts
  • Key lifecycle: expiry reminders, new requests
  • Enable/disable individual alert types
  • Set alert recipients

Automated Backend

All cloud resources are managed programmatically. No manual console interaction required after initial setup.

Key Provisioning Auto

  • Automatically create credentials when key is approved
  • Tag resources: cost centre code, developer, created date, expiry
  • Target: under 10 seconds

Access Control Auto

  • Restrict keys to specific models per configuration
  • Enforce model restrictions at cloud layer (not just in app)

Key Lifecycle Auto

  • Automatic expiry enforcement
  • Revocation: clean up all cloud resources
  • Rolling period: sliding window, oldest spend falls off
  • SSO offboarding: keys auto-disabled when user removed from identity provider
  • Developer revocation: immediate deactivate + flag for cleanup

Cost & Budget Enforcement Auto

  • Track cost per key, per developer, per cost centre (in dollars)
  • Cost calculated from token consumption + model pricing (from AWS)
  • Three enforcement levels (all apply simultaneously):
  • → Per-key rolling period: only that key stopped
  • → Per-key lifetime budget: key stopped permanently
  • → Cost Centre Budget Cap: all cost centre keys stopped
  • Soft alerts at configurable thresholds
  • Per-key rolling: auto re-enable when spend falls outside window
  • Cost centre cap: re-enable when CCO increases budget
  • Developer notified on stop/re-enable

Guardrails Future

  • Content filtering per cost centre or globally
  • Configurable by admin

Non-Functional Requirements

Quality attributes and operational requirements.

🔒 Security

  • All keys encrypted at rest and in transit
  • Keys retrievable after creation (secured by authentication)
  • Audit log of all key operations — retained indefinitely
  • Audit logs accessible to Administrators only
  • Cloud-level audit trail integration
  • Role-based access control (RBAC)
  • OWASP Top 10 compliance

📈 Scalability

  • Support ~300 developers and 1000s of keys
  • Async key provisioning if needed

🔄 Reliability

  • Handle cloud API failures gracefully (retry with backoff)
  • Reconciliation to detect drift between app state and cloud state

👁️ Observability

  • Application logging and monitoring
  • Alerts on failed provisioning, expired keys, budget thresholds
