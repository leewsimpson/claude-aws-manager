01 — Capabilities

Govern access without becoming a bottleneck.

Everything an organisation needs to put Claude Code in developers' hands today — and pull it back the second a budget tips over. No proxy, no inference middleware, no token vault to defend.

/ 01

Self-service requests

Developers request a key against a cost centre with optional justification. Cost-centre owners approve with per-request constraints — or auto-approve themselves. State changes flow through a single audit log.

flow: request → approve → provision
/ 02

Bedrock API Keys, not raw IAM

Each key creates a backing IAM user claude-{dev}-{cc} with a Bedrock service-specific credential. Tokens are revealed once at provisioning and never stored. Lost? Regenerate.

aws: CreateServiceSpecificCredential
/ 03

Model allow-lists at the IAM layer

Approved models become an inline IAM policy scoping bedrock:InvokeModel to specific model ARNs. Enforcement is the same primitive AWS uses — not an app-side guardrail you have to trust.

scope: ARN-level, region-pinned
/ 04

Hard budget enforcement

A background poller computes spend from CloudWatch token metrics every 2 minutes and deactivates the key when rolling, lifetime, or cost-centre budgets are crossed. Re-enables automatically as the window clears.

limit: rolling · lifetime · CC budget
/ 05

Cost-centre governance

Cost centres own budgets, key holders, and inference profiles. CCOs see real-time spend across all keys they're accountable for; admins see the whole organisation. One CC per developer per model lane.

roles: Developer · CCO · Admin
/ 06

Inference profiles per CC × model

Each cost centre gets dedicated Bedrock inference profiles per approved model — so CloudWatch metrics roll up cleanly per CC and per model, without parsing logs or paying for Cost Explorer queries.

cloudwatch: profile-level metrics
/ 07

App-calculated cost, cached pricing

Spend is computed from token counts and a cached pricing_cache table — deterministic, reproducible, and not dependent on Cost Explorer's day-late aggregates.

precision: Decimal, never floats
/ 08

Mock & real AWS, one switch

All AWS calls flow through an AwsService abstraction. AWS_MODE=mock for local dev with no credentials; flip to real for production. App code never touches boto3.

dev: no AWS account required
/ 09

Append-only audit log

Every approval, provision, stop, restart, and revoke is recorded with actor, IP, and JSON metadata. Admin-only access. Useful for compliance reviews — and indispensable when something goes sideways.

retention: append-only, indexed
02 — How it works

From request to revoked, in one append-only timeline.

STEP 01

Request

Developer picks a cost centre, adds an optional justification, and submits a key request.

STEP 02

Approve

The cost-centre owner approves with constraints — models, rolling limit, lifetime budget, expiry. Or auto-approves if it's their own request.

STEP 03

Provision

IAM user, inline policy, and Bedrock service credential are created in one transactional flow. The bearer token is revealed once. Failures roll back AWS state.

STEP 04

Govern

Usage flows in from CloudWatch. Spend ticks against rolling, lifetime, and CC budgets. The poller stops or restarts the key automatically. Every step audited.

04 — Architecture

Boring stack. Sharp boundaries.

FastAPI + Postgres + React. Bedrock as the credential plane. CloudWatch as the meter. The platform stays out of the inference path on purpose.

  • BackendPython 3.12 · FastAPI · SQLAlchemy + Alembic · psycopg v3 · uv
  • FrontendReact + Vite + TypeScript · TanStack Query · React Router
  • DatabasePostgreSQL 16 · JSONB for policies, approval constraints & audit metadata
  • AWSBedrock API Keys · CloudWatch metrics · IAM inline policy · ap-southeast-2
  • DeployDocker Compose locally · ECS + RDS/Aurora in production
  • AuthJWT for the PoC · OIDC/SAML SSO planned for production
